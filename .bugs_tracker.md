## ✅ Resolved Bugs

### BUGS-005: Missing spaCy Model Data
**Status:** Resolved | **Severity:** Critical (Blocker)
**Description:** Application crashed with `[E050] Can't find model 'en_core_web_sm'`.
**Root Cause:** The `spacy` library was installed, but the model data itself was never downloaded into the Docker image.
**Fix:** Added a `RUN python -m spacy download ${SPACY_MODEL}` command to the `Dockerfile`.

### BUGS-004: Container Crash on Startup (Multiple Causes)
**Status:** Resolved | **Severity:** Critical (Blocker)
**Description:** The `ui-1` container was crashing due to a chain of initialization failures.
**Root Causes & Fixes:**
  - `ModuleNotFoundError: 'app'`: Fixed by setting `ENV PYTHONPATH=/app` in `Dockerfile`.
  - `ValueError: Extra inputs not permitted`: Fixed by adding `extra='ignore'` to Pydantic `AppConfig`.
  - `ModuleNotFoundError: 'msal'`: Fixed by adding `msal` to `pyproject.toml`.
  - `ImportError: Circular Import`: Fixed by refactoring `app/pm_tools/__init__.py`.
  - `OperationalError: could not translate host name "db"`: Fixed by correcting `DATABASE_URL` to `postgresql+psycopg2://...`.
  - `AttributeError: 'dict' object has no attribute 'max_turns'`: Fixed by making `AppConfig` fully type-safe with Pydantic models.
  - `ImportError: No module named 'MockPMAdapter'`: Fixed typo in `PMToolFactory`.

## 🔍 Bug Patterns Identified (AI-UPDATED)

### Pattern 1: Build vs. Runtime Dependencies
**Occurrences:** 1 time (spaCy model)
**Root cause:** Assuming that installing a library (`spacy`) also provisions its required data files.
**Solution:** The `Dockerfile` must be responsible for installing libraries (via `pyproject.toml`) AND downloading any required data (models, assets, etc.).

### Pattern 2: Ambiguous Configuration
**Occurrences:** 1 time (Database URL)
**Root cause:** Using a generic connection string (`postgresql://`) that allows a dependency (SQLAlchemy) to guess the driver.
**Solution:** Always use explicit configurations (e.g., `postgresql+psycopg2://`) to remove ambiguity.

### Pattern 3: Incomplete Pydantic Models
**Occurrences:** 1 time (`AppConfig`)
**Root cause:** Defining nested configuration sections as generic `dict` types in Pydantic, which prevents validation and attribute access.
**Solution:** Every level of a configuration file should be represented by a specific Pydantic `BaseModel`.
